Value Object Generation

The definition and therefore the generation of a ValueObject type of class is done via a Java definition interface. This interface has a name ending in ef and extends directly or indirectly the org.highway.vo.ValueObject interface. For example, the mypackage.MyValueObjectDef package defines the mypackage.MyValueObject class.

The ValueObject type of class generated by vogen will implement the ef interface it defines. For example, the mypackage.MyValueObject class will implement the mypackage.MyValueObjectDef interface
NB: the class generated implements all the interfaces the ef interface inherits at the same time.

There are two ways of defining the value object properties: the constants and the getters. The constants method is deprecated and will be eliminated in version 1.3. The getters method is new and must be generalized.

The interface java.lang.String type constants will define the value object properties. The constant name must be the name of the property in capital letters with characters underlined to separate the words. The property value must be the precise name of the property and must therefore observe the JavaBeans naming convention. For example:

  String FIRST_NAME = "firstName"; // defines the JavaBean property firstName

The interface JavaBean getter type of method will define the value object properties. For example:

  String getFirstName(); // defines the JavaBean property firstName

There are two ways of having a value object generated by vogen inherited. You can use the highway.vo.superclass tag and therefore give the name of the class to be extended. You can also use inheritance in the definition interfaces. If ADef extends BDef, the A class will extend the B class. If both methods are used the tag has priority. If the developer does not specify the super class, the class generated extends org.highway.vo.ValueObjectAbstract.

The following table describes the javadoc tags that enable you to define the value object characteristics and its properties. The class type tags are defined at interface level and define the value object overall characteristics. The property type tags are defined at constants level and define the characteristics for the value object properties, which these constants represent.

Inheritance

Several types of mapping are supported in the case of an inheritance relationship:

- table-per-class-hierarchy: a single table, with a discriminatory column

- table-per-subclass: one table per class (one for the super-class and one per sub-class)

- table-per-concrete-class: one table per sub-class (concrete class), but not for the (abstract) super-class

table-per-class-hierarchy

The properties of the super-class and the sub-classes are all mapped on a single table. A discriminatory column is added to determine the class corresponding to a given line.

The highway.mapping tag must be positioned in the super-class as well as the sub-classes, with the discriminator-value parameter filled in.
The highway.mapping.discriminator tag, which enables you to specify the discriminatory column only appears in the super-class.

Limitation: the columns defined at sub-class level cannot have any NON-ZERO constraints.

E.g.:

/**
 * @highway.mapping table="EMPLOYE"
 *                discriminator-value="0"
 * @highway.mapping.discriminator type="char"
 *                              column="TYPE"
 */
public interface EmployeDef extends ValueObject {...}
 
/**
 * @highway.mapping discriminator-value="1"
 */
public interface CadreDef extends EmployeDef {...}
 
/**
 * @highway.mapping discriminator-value="2"
 */
public interface TechnicienDef extends EmployeDef {...}

table-per-subclass

Each class whether super-class or sub-class is mapped on a different table.
The ID column defined at super-class level is added to the sub-class tables. The link between the super-class and the sub-classes is controlled via this ID; a discriminatory column is not required.

The highway.mapping tag must be positioned in the super-class and the sub-classes, but this time without any discriminator-value parameter.
The highway.mapping.discriminator tag is not used.

Limitation: in the current version of Socle, the identifier defined at super-class level cannot be made up of several columns; it must necessarily be a simple identifier.

E.g.:

/**
 * @highway.mapping table="EMPLOYE"
 */
public interface EmployeDef extends ValueObject {...}
 
/**
 * @highway.mapping table="CADRE"
 */
public interface CadreDef extends EmployeDef {...}
 
/**
 * @highway.mapping table="TECHNICIEN"
 */
public interface TechnicienDef extends EmployeDef {...}

table-per-concrete-class

This approach can be envisaged when the super-class is abstract. A table is used for each sub-class, but not for the super-class. The super-class properties are mapped on the sub-class tables.

The highway.mapping tag is not positioned in the super-class, but only in the sub-classes. On the other hand, the highway.mapping.id and highway.mapping.property tags can be used in the super-class.
The highway.mapping.discriminator tag is not used.

Limitations:

·         By definition, the super-class is necessary abstract.

·         Polymorphic loading and joins are not supported.
For example, a (super-class) employee cannot be loaded as follows: select(Employe.class, id)

E.g.:

public interface EmployeDef extends ValueObject
{
    /**
     * @highway.mapping.id column="ID"
     */
    Long getId();
}
 
/**
 * @highway.mapping table="CADRE"
 */
public interface CadreDef extends EmployeDef {...}
 
/**
 * @highway.mapping table="TECHNICIEN"
*/
public interface TechnicienDef extends EmployeDef {...}
